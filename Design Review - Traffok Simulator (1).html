Design Review · April 2026

Tokai Traffic Simulator
UI Design Critique

Reviewed against the Kinetic Sentinel design spec — audience: community residents
Source: App.jsx · 42kb Spec: DESIGN.md · Kinetic Sentinel 5 Critical Findings 3 Moderate Findings Audience: Community / Residents
Overview

Strong foundation, spec compliance gaps

The simulator's data model and interaction logic are solid — scenario switching, what-if toggles, and the LOS calculation engine all work well. The design intent (dark control-center aesthetic) is clear. However, the implementation diverges from the Kinetic Sentinel spec in several structural ways, and the interface assumes a level of technical fluency that community residents may not have. The findings below are prioritised for maximum impact with minimal refactoring.

Finding 01 — Critical

The "No-Line Rule" is not followed

The spec explicitly prohibits 1px solid borders for layout containment. The current app uses them extensively — on every card, the sidebar, the header, zone counter rows, and what-if group panels. This creates a "wireframe" quality that undercuts the depth the spec is trying to achieve.

Remove all structural border declarations
Critical
The spec says: "Boundaries must be defined solely through background color shifts, tonal transitions, or negative space." Currently ~40+ instances of border: 1px solid exist for layout containment. Only the "Ghost Border" fallback (for inputs) and the left-accent strips on zone counters should remain.
Recommendation Replace structural borders with a one-step darker background — e.g. swap #0f1117 cards on a #0a0e1a sidebar by bumping the card to surface_container_high (#222a3d) instead. The contrast does the separation work.
Finding 02 — Critical

Glassmorphism is completely absent

The spec's "Glass & Gradient Rule" defines the visual identity of the product. The HUD overlays, floating status indicators, and the "WHAT IF... ACTIVE" badge should all use the glassmorphism recipe. Currently every element is fully opaque.

Flat fill, solid border — feels like a chip, not a HUD
Recommended — Glass HUD
System Override Active
2 what-if groups enabled
backdrop-filter: blur(24px), 70% opacity surface
Apply glassmorphism to floating / overlay elements
Critical
The spec's recipe: surface_container at 60–80% opacity + backdrop-filter: blur(20–40px). Apply to: (1) the "WHAT IF ACTIVE" status banner, (2) the "WHAT IF ACTIVE" map overlay text, (3) the intersection tooltip labels on hover, (4) the LOS legend panel on the map.
Recommendation background: rgba(34, 42, 61, 0.72); backdrop-filter: blur(24px); This one CSS change transforms the flat overlays into the layered HUD feel the spec describes. Combine with a very subtle inset shadow: box-shadow: inset 0 1px 0 rgba(218,226,253,0.06).
Finding 03 — Critical

Resident audience cannot parse the technical labels

The audience is community residents, not traffic engineers. Labels like "v/c ratio 1.25", "LOS F", "VHT", and "peakMin" are presented without plain-language explanation. The what-if toggle items have good descriptions but they're buried in collapsed accordions.

Add plain-English status callouts above the technical data
Critical
Before any metric card, give residents a one-line verdict in plain language. "LOS F — this road will be at breaking point" is far more useful to a resident than a raw v/c ratio. The technical data can still show below for anyone who wants it.
Pattern to implement Add a "verdict banner" at the top of the sidebar (above Layer 1) that reads, e.g.: "Under the TIA's own worst-case figures, Ruskin Rd will operate beyond capacity every morning. The TIA never modelled this road." Update it reactively as scenario changes.
Expand what-if groups by default (or one of them)
Moderate
All three what-if groups are collapsed on load. A resident opening the tool for the first time has no idea these controls exist. At least one group — Road Capacity Realities — should be expanded by default to invite exploration.
Recommendation Set const [expanded, setExpanded] = useState(group.id === 'road_capacity') in WhatIfGroup. Also add a subtle pulsing dot on collapsed groups that have items not yet seen.
Finding 04 — Critical

Emoji in the header violates spec + reads as informal

The header uses "🏫 Tokai High School" and the school map marker uses a school emoji. The spec's Director's Note says "every pixel should feel like it was placed by a system that knows more than the user" — emoji undercut that authority, especially for a planning context where credibility matters to residents.

Replace emoji with geometric SVG markers and text labels
Critical
The header title, the school map marker, and the what-if group icons (🚗, 📋, 🚶) all use emoji. These feel consumer-grade and inconsistent with the control-centre aesthetic.
Recommendation For the header: remove the emoji, let the typography carry the weight — "Tokai High School — Traffic Impact Simulator" in Space Grotesk 700 needs no decoration. For the school map marker: use a filled square with a bold "HS" monogram in primary colour. For what-if icons: use simple SVG glyphs (car outline, clipboard, walker silhouette) at 14px.
Finding 05 — Critical

The header font uses DM Mono — spec calls for Space Grotesk + Inter

The entire app uses DM Mono as its font family, including the main title and all UI text. The spec defines a two-font system: Space Grotesk for display/headlines (the "big data" indicators), Inter for labels and body. DM Mono is a code-editor font — fine for telemetry readouts, but using it for everything creates a monotone typographic texture.

Apply the two-font hierarchy from the spec
Moderate
DM Mono should be reserved for: v/c ratios, LOS codes, numeric readouts, code-like labels. Space Grotesk should own: the app title, scenario labels, section headers, zone names. Inter should handle: descriptions, what-if item copy, footer disclaimers.
Font assignment map
Space Grotesk 700 → App title, scenario H/M/L, section eyebrows

Inter 400/500 → Descriptions, what-if copy, footer

DM Mono 400 → v/c values, LOS codes, %, queue counts
Finding 06 — Moderate

The map has no interactivity or hover states

The SVG road map is static. Clicking on a road segment, intersection marker, or school icon does nothing. For a community audience exploring the map, hover tooltips with plain-English explanations of what each road is and why it matters would significantly improve comprehension.

Add hover/click tooltips to road segments and intersection markers
Moderate
Each road polyline and intersection circle should respond to hover with a glass HUD tooltip showing: road name, current LOS in plain English, capacity vs load, and whether it was studied in the TIA. This turns the map from a static diagram into an exploration surface.
Recommendation Implement a hoveredSeg state in the parent. On onMouseEnter per polyline, show a glassmorphic tooltip positioned near the cursor using a foreignObject or an absolutely-positioned div above the SVG. Include a "Why does this matter?" line written for residents.
Finding 07 — Moderate

Scenario descriptions are too technical for residents

The scenario description text (e.g. "TIA worst-case — 840 trips, sharp 45-min peak, no reductions applied") is written for planners. Residents need to understand what that means for their morning.

Add resident-facing scenario summary text
Moderate
Each SCENARIOS entry should include a second description field written in plain English. Display it prominently, with the technical description collapsed or shown in a smaller muted style below.
Suggested copy additions to SCENARIOS data

H: "This is the council's own worst-case projection — 840 cars arriving within 45 minutes, with no mitigations. This is the baseline the TIA submitted to council."

M: "A 15% reduction in car trips — possible if a School Travel Plan is in place. Still significantly exceeds road capacity."

L: "A 40% reduction — only achievable with active modal shift. The TIA made no commitment to achieving this."
Finding 08 — Positive

What's working well

The data model is excellent
Keep
The LOS calculation engine, capacity modifiers, and delta tracking in calcVC() are well-structured and accurate. The what-if system layering on top of the baseline is the right architecture. The SVG road network is also a strong foundation for interactivity.
Color-coded LOS system is clear and consistent
Keep
The green → yellow → orange → red → dark-red gradient for LOS A through F is immediately legible. The glow effect on road segments (shadow polyline + main polyline) gives the map a professional depth that already leans into the spec's neon aesthetic. Keep and strengthen this.
The left-accent stripe on zone counters is the right pattern
Refine
Using a coloured left border as an accent (not a structural border) is spec-compliant. This is the correct way to use colour emphasis in the design system. It should be extended: remove the surrounding container border but keep and intensify the left stripe with a matching subtle glow (box-shadow: -2px 0 12px color).

Closing Note

The design system's intent is clear — close the gap

The Kinetic Sentinel spec is well-written and has a strong aesthetic point of view. The implementation just hasn't caught up to it yet. The six changes above are all achievable without touching the data model or layout structure — they're purely visual and copy refinements that would significantly close the gap between what was designed and what was built.

Every pixel should feel like it was placed by a system that knows more than the user. Keep it dark, keep it glowing, and keep it deep.

— DESIGN.md Director's Note, Kinetic Sentinel spec
